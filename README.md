# Асинхронный сервис управления задачами

Асинхронный, масштабируемый и отказоустойчивый сервис обработки задач. Задачи создаются через REST
API, помещаются в очередь RabbitMQ согласно приоритету, обрабатываются в фоне параллельными
воркерами, а их полный жизненный цикл сохраняется в PostgreSQL.

## Обзор

- **Создание** задачи через REST → сохраняется как `NEW` → публикуется в приоритетную очередь →
  помечается `PENDING`.
- **Воркеры** читают сообщения конкурентно, переводят задачу `IN_PROGRESS → COMPLETED/FAILED` и
  сохраняют результаты/ошибки.
- **Клиенты** опрашивают состояние задачи через REST (`GET .../{id}` или `.../{id}/status`) и могут
  отменять задачи, пока те находятся в статусе `NEW`/`PENDING`.

### Архитектура

Чистые зависимости, направленные внутрь (SOLID):

```
                +-------------------+
   HTTP  --->   |  FastAPI (api)    |
                +---------+---------+
                          |
                          v
                +-------------------+        publish (aio-pika)
                |  TaskService      | ----------------------------+
                | (бизнес-правила)  |                             |
                +---------+---------+                             v
                          |                              +-----------------+
                          v                              |   RabbitMQ      |
                +-------------------+                     | приоритетная    |
                |  TaskRepository   |                     |    очередь      |
                +---------+---------+                     +--------+--------+
                          |                                       |
                          v                                       | consume
                +-------------------+                             v
                |   PostgreSQL      |                     +-----------------+
                |  (таблица tasks)  | <-------------------|   Worker(s)     |
                +-------------------+  сохранение         |  TaskProcessor  |
                                       жизненного цикла   +-----------------+
```

Слои: `api → service → repository → model/db`, причём сервис зависит от *абстракции* публикатора
(`AbstractTaskPublisher`), поэтому транспорт и хранилище взаимозаменяемы (например, `NullPublisher`
и SQLite в тестах).

### Поток запроса и обработки

1. `POST /api/v1/tasks` валидирует тело запроса и вызывает `TaskService.create_task`.
2. Сервис сохраняет задачу как `NEW`, переводит в `PENDING` и **коммитит эту строку до публикации**,
   затем публикует `{ "task_id": ... }` в долговечную приоритетную очередь (`x-max-priority=10`,
   приоритет сообщения выводится из приоритета задачи). Коммит до публикации исключает гонку
   «публикация раньше коммита», когда воркер мог бы прочитать сообщение прежде, чем строка станет
   видимой. Если публикация не удалась, задача не остаётся молча в очереди — ошибка возвращается
   как `503`.
3. Воркер получает сообщение (prefetch = `WORKER_CONCURRENCY`), открывает собственную сессию БД и
   выполняет `TaskProcessor.process`: пропуск, если `CANCELLED`, иначе `mark_in_progress` (чтение
   под блокировкой строки `FOR UPDATE`, возвращает `None` для любого статуса, кроме `PENDING`, что
   делает повторную доставку идемпотентной), выполнение работы, затем `mark_completed`
   (с результатом) или `mark_failed` (с ошибкой).
4. Сообщения **подтверждаются (ACK) только после** завершения обработки (успех либо обработанная
   бизнес-ошибка). При временной/инфраструктурной ошибке воркер **не** делает NACK/requeue: он
   читает заголовок `x-retry-count` и, пока число попыток меньше предела
   (`MAX_DELIVERY_ATTEMPTS = 3`), **переопубликовывает новое persistent-сообщение** (с тем же
   приоритетом) с увеличенным счётчиком и подтверждает оригинал. Поскольку ACK следует за успешной
   обработкой, воркер, упавший в середине задачи, оставляет сообщение неподтверждённым, и брокер
   **передоставляет** его другому воркеру.
5. Когда `x-retry-count` достигает предела, воркер делает `reject(requeue=False)`, и сообщение
   **отправляется в dead-letter** через DLX → DLQ; некорректные/непарсящиеся сообщения отправляются
   в dead-letter сразу. Это не даёт одному «ядовитому» сообщению зациклиться навсегда и
   заблокировать очередь.

## Технологический стек

- Python 3.11+
- FastAPI + Uvicorn
- SQLAlchemy 2.0 (async) + asyncpg
- Alembic (асинхронное окружение миграций)
- aio-pika (RabbitMQ)
- pydantic v2 + pydantic-settings
- PostgreSQL 14, RabbitMQ 3 (management)
- pytest, pytest-asyncio, httpx, aiosqlite (тесты)

## Конфигурация

Все настройки берутся из переменных окружения (или файла `.env`). Скопируйте шаблон:

```bash
cp .env.example .env
```

| Переменная           | Описание                                       | Значение по умолчанию в Compose                              |
|----------------------|------------------------------------------------|-------------------------------------------------------------|
| `DATABASE_URL`       | DSN для async SQLAlchemy/asyncpg                | `postgresql+asyncpg://postgres:postgres@postgres:5432/tasks` |
| `RABBITMQ_URL`       | URL подключения AMQP                            | `amqp://app:app@rabbitmq:5672/`                             |
| `TASK_QUEUE_NAME`    | Имя приоритетной очереди                        | `tasks`                                                     |
| `WORKER_CONCURRENCY` | Prefetch воркера / число параллельных задач     | `4`                                                        |
| `APP_NAME`           | Имя приложения                                 | `async-task-service`                                       |
| `LOG_LEVEL`          | Уровень логирования                            | `INFO`                                                     |

RabbitMQ работает под отдельной учётной записью приложения (`RABBITMQ_DEFAULT_USER` /
`RABBITMQ_DEFAULT_PASS`), а не под встроенным пользователем `guest`, и `RABBITMQ_URL` должен
использовать те же учётные данные. Значения из `.env.example` предназначены **ТОЛЬКО ДЛЯ
РАЗРАБОТКИ** и должны быть заменены на надёжные секреты в любом общем или промышленном окружении.

## Запуск через Docker Compose

```bash
cp .env.example .env
docker compose up --build
```

Это поднимает `postgres` и `rabbitmq` (оба с healthcheck), запускает одноразовый сервис `migrate`
(`alembic upgrade head`) после того, как база станет healthy, а затем стартует `api` и `worker`.

- API: http://localhost:8000 (документация OpenAPI на `/docs`)
- Панель управления RabbitMQ: http://localhost:15672

**Отказоустойчивость, встроенная в Compose:**

- **Политики перезапуска:** `postgres`, `rabbitmq`, `api` и `worker` запускаются с
  `restart: unless-stopped`, поэтому восстанавливаются после падений и перезагрузок хоста.
  Одноразовый сервис `migrate` использует `restart: "no"` (он должен выполниться ровно один раз).
- **Healthcheck для API:** сервис `api` проверяется через `GET /health` (лёгкая проверка
  живости, возвращающая `{"status": "ok"}`, без обращения к БД), что позволяет Docker отслеживать
  состояние контейнера и реагировать на него.
- **Долговечное состояние брокера:** данные RabbitMQ хранятся на именованном томе `rabbitmqdata`,
  смонтированном в `/var/lib/rabbitmq`, поэтому долговечные очереди и persistent-сообщения
  переживают пересоздание контейнера (а не только внутренние перезапуски). Данные PostgreSQL
  сохраняются на томе `pgdata`.

### Масштабирование воркеров

Воркеры не хранят состояние и читают из общей очереди, поэтому мощность наращивается запуском новых
контейнеров:

```bash
docker compose up --scale worker=4
```

Каждый воркер дополнительно обрабатывает до `WORKER_CONCURRENCY` сообщений параллельно, так что
эффективный параллелизм равен `число_воркеров × WORKER_CONCURRENCY`.

## Миграции

Миграции находятся в `alembic/` и подключены к **асинхронному** окружению (`alembic/env.py`),
которое читает `DATABASE_URL` из `Settings` и использует `Base.metadata` как целевую метаданную.
Compose-сервис `migrate` применяет их автоматически. Для запуска вручную:

```bash
# внутри окружения проекта
alembic upgrade head      # применить
alembic downgrade -1      # откатить последнюю миграцию
alembic revision --autogenerate -m "message"   # создать новую миграцию
```

## Эндпоинты API

Префикс роутера: `/api/v1/tasks` (тег `tasks`).

| Метод    | Путь                             | Описание                     | Успех   | Ошибки        |
|----------|----------------------------------|------------------------------|---------|---------------|
| `POST`   | `/api/v1/tasks`                  | Создать задачу               | `201`   | `422`         |
| `GET`    | `/api/v1/tasks`                  | Список (фильтр + пагинация)  | `200`   | `422`         |
| `GET`    | `/api/v1/tasks/{task_id}`        | Получить одну задачу         | `200`   | `404`         |
| `DELETE` | `/api/v1/tasks/{task_id}`        | Отменить задачу              | `200`   | `404`, `409`  |
| `GET`    | `/api/v1/tasks/{task_id}/status` | Получить статус задачи       | `200`   | `404`         |

Параметры запроса для списка: `status?`, `priority?`, `limit` (1..100, по умолчанию 20),
`offset` (>=0, по умолчанию 0).

## Масштабируемость и отказоустойчивость

- **Горизонтальное масштабирование:** API и воркеры — контейнеры без состояния, масштабируются
  независимо. Воркеры конкурируют за сообщения в одной долговечной очереди, что даёт автоматическую
  балансировку нагрузки.
- **Приоритетное планирование:** очередь RabbitMQ с `x-max-priority` обрабатывает `HIGH` раньше
  `MEDIUM`/`LOW`.
- **Долговечность:** долговечная приоритетная очередь (`x-max-priority=10`, `x-dead-letter-exchange`)
  + persistent-сообщения переживают перезапуски брокера; долговечная DLQ, привязанная к fanout-DLX,
  принимает dead-letter сообщения. Жизненный цикл задачи — источник истины в PostgreSQL.
- **Доставка at-least-once:** ACK отправляется только *после* завершения обработки, поэтому падение
  воркера в середине задачи оставляет сообщение неподтверждённым, и брокер передоставляет его
  другому воркеру. Обработанные бизнес-ошибки фиксируются как `FAILED` и подтверждаются;
  действительно отсутствующая задача пропускается и подтверждается. Повторная доставка
  **идемпотентна** — `mark_in_progress` читает строку `FOR UPDATE` и завершается рано для любого
  статуса, кроме `PENDING`, а `CANCELLED`-задачи пропускаются — и строка `PENDING` **коммитится до
  публикации**, поэтому воркер никогда не видит ещё не сохранённую задачу.
- **Ограниченные повторы + обработка «ядовитых» сообщений:** временные/инфраструктурные ошибки
  **не** обрабатываются через NACK/requeue. Вместо этого воркер читает заголовок `x-retry-count` и,
  пока число попыток меньше `MAX_DELIVERY_ATTEMPTS` (3), **переопубликовывает новое
  persistent-сообщение** (сохраняя приоритет) с увеличенным счётчиком и подтверждает оригинал. По
  достижении предела — или для некорректного/непарсящегося сообщения — воркер делает
  `reject(requeue=False)`, и сообщение **отправляется в dead-letter** через fanout-DLX в долговечную
  DLQ, так что одна плохая задача не зацикливается навсегда и не блокирует очередь.
- **Живость:** API предоставляет `GET /health`, возвращающий `{"status": "ok"}`; compose-сервис
  `api` использует его как Docker healthcheck.
- **Backpressure (контроль нагрузки):** `prefetch_count = WORKER_CONCURRENCY` ограничивает объём
  работы в обработке на одного воркера.
- **Корректное завершение:** воркеры прекращают чтение и дорабатывают текущие задачи по
  `SIGTERM`/`SIGINT`.
- **Старт по готовности:** compose блокирует `api`/`worker` до завершения миграций и до тех пор,
  пока зависимости не сообщат о готовности (healthy).
